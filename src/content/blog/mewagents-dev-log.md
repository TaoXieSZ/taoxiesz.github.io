## 一、端到端验证

### 任务流转

一个写作任务从发起到完成，经过 5 个 agent 接力，约 2 分钟：

```
阿咪(分拣) → 策划喵(规划) → 审核喵(审核) → 调度喵(派发) → 执行 → Done
```

运行模式：轻量模式（`dashboard/server.py` + JSON 文件，无 PostgreSQL/Redis）。

### 实际流转记录

```
JJC-20260401-001  "撰写AI agent多智能体协作短文"

09:35:19  主人 → 策划喵    阿咪整理任务小纸条
09:36:06  策划喵 → 审核喵   短文方案已提交审议
09:36:45  策划喵 → 调度喵   审核通过，转调度喵派发
09:37:14  调度喵 → 主人     ✅ 完成
```

Dashboard 实时同步正常，todo 进度条正确更新。

### 环境修复

跑之前发现三个问题，全部通过 `bash install.sh` 一键修复：

| 问题 | 原因 |
|------|------|
| 只有 3 个 agent 注册 | mewagent 团队的 11 个没执行过注册 |
| agent 读不到 tasks_source.json | data 软链接缺失 |
| agent 调用报 401 | API Key 没同步到各 agent |

修复后额外发现：默认模型配的是 `anthropic/claude-opus-4-6`，但没有 Anthropic key，改成了 `openai-codex/gpt-5.4`。

## 二、踩过的坑

### 坑 1：ami 跳过协作流程

发任务 "帮我写一篇500字短文"，ami 直接自己写了返回，没建 JJC 任务、没转策划喵。

**诊断**：让 ami 复述 SOUL.md 规则 → 能完整复述 → 说明读到了但没遵守。

**根因**：gpt-5.4 的 "helpful override" —— 模型判断自己能做就直接做了，绕过 SOUL.md 的协作约束。

**修复**：在 SOUL.md 加了硬性禁止规则：

```
铁律：你是分拣员，不是执行者！
你绝对不能自己执行任何正式任务。
即使你觉得自己能写出来，也必须走任务流程。
```

加了之后配合消息里的显式提醒，ami 正确走了完整流程。

**结论**：对 LLM agent 来说，"绝对不能做 X" 比 "应该做 Y" 有效得多。引导规则不够，要用禁止规则。

### 坑 2：审核喵秒过

审核喵没有表现出实质审查行为，直接 approve 了。

**后续测试**：用故意模糊的需求（"重构整个项目的数据库层"）做驳回测试。

```
JJC-20260401-002  "重构整个项目的数据库层"

12:08:23  主人 → 策划喵    驳回测试：故意模糊的需求
12:09:23  策划喵 → 审核喵   数据库层重构方案提审
12:20:49  审核喵 → 策划喵   ❌ 打回：缺方案正文、分阶段迁移/回滚、验收指标与资源排期
12:21:57  策划喵 → 审核喵   数据库层重构执行方案提审（补充后重新提交）
12:23:15  策划喵 → 调度喵   审核通过，转派执行
12:24:23  调度喵 → 主人     ✅ 完成
```

审核喵在第一次提交时正确驳回了不完整的方案，策划喵补充后二次提交才通过。**驳回功能验证通过**。

### 架构设计

创建了 `Icon` 和 `IconOrEmoji` 双模式组件：

- **`Icon`**：接受 icon key 字符串，渲染对应 Lucide SVG
- **`IconOrEmoji`**：兼容后端返回的 emoji 数据 — 能匹配 Lucide key 就用 SVG，否则原样渲染 emoji 文字

这样 store 默认值用 icon key，后端数据用 emoji，两套数据源无缝兼容。

### 猫咪姿势组件

为候命中的喵员创建了 `CatPose` 组件，包含 7 种手绘 SVG 猫咪姿势：

| 姿势 | 描述 |
|------|------|
| 睡觉猫 | 蜷成一团 + zzz |
| 洗脸猫 | 抬起前爪擦脸 |
| 伸懒腰猫 | 前肢前伸 |
| 趴着猫 | 肚子贴地 |
| 打滚猫 | 肚皮朝上 |
| 蹲坐猫 | 端正坐姿 |
| 歪头猫 | 好奇歪头 |

基于 agent id 哈希稳定分配 — 同一个喵员每次看到的姿势一样，不同喵员姿势不同。
