今天在 agent-tickets 上发生的事,值得单独记一篇:一个修 bug 的请求,顺嘴带出一句感慨,被 agent 反向拷问成了一个功能,当天上线。

## 起点:每天 16:00 都来,但点了没反应

agent-tickets 有个 launchd 定时任务,每天 16:00 跑 `ticket report --notify`——日报写盘、看板刷新、macOS 通知弹出。功能一直在正常跑,唯独一件事很膈应:**通知点了没反应**,跳不到工单系统。

我以为是配置问题,让 agent 找一下。结论是:根本不是配置,是 **macOS `display notification` 的硬限制**——osascript 发的通知天生不支持挂点击动作,点击只会激活脚本编辑器。怎么配都没用,只能换工具。

修法本身不复杂:换 `terminal-notifier`,带 `-open file://…/dashboard.html`,点击直达静态看板。但有个陷阱值得记下来:

> **launchd 的 PATH 里没有 `/opt/homebrew/bin`。**交互 shell 里 `shutil.which("terminal-notifier")` 好好的,第二天 16:00 定时跑的时候会静默找不到、退回旧行为——你以为修好了,其实只在你自己的终端里修好了。定时任务环境和交互环境是两个世界,兜底路径必须显式写。

## 顺嘴一句话,被拷问了四题

修完之后我顺嘴问了句「项目里的 backlog 有哪些」,又补了句感慨:**vibecoding 时代,最怕的就是没有 todo 去做**——这本身就该是个功能。然后加了个 `/grill-me`,让 agent 别顺着我,拷问我。

它没有直接开写代码,先去查了事实,然后甩回来第一个问题就打在要害上:

> 你说「最怕没有 todo 去做」——但 `ticket list` 里明明躺着 9 个 0% 的 ready 单。数据不缺,缺的到底是哪一环?

四轮问下来,每题都是选择题、带推荐项:

1. **痛点在哪一环?** 视图/颗粒度/供给/决策——选了**决策环**:单子都在,但坐下来要自己从 22 行列表里挑,认知负担在这。
2. **拿什么规则挑?** 我当时的局面:1 个聚焦单、3 个 80%+ 的半成品、16 个停滞单、9 个 0% 新坑。选了**收尾优先**——vibecoding 真正的坑不是没新坑可挖,是半成品越积越多。
3. **答案从哪冒出来?** 做在没人看的地方等于没做。选了**三处都要**:CLI、看板置顶卡、日报建议。
4. **项目维度做多深?**「该做什么」和「还有什么」是两个问题,拆成 `ticket next [repo]` 和 `ticket backlog <repo>` 双命令。

最值钱的不是这四个答案,是拷问前的那次**事实检查**:agent 翻了状态库和日报生成代码,发现 backlog 数据早就全量入库,甚至**建议引擎都已经存在**(日报里那行「建议 3」),只是只做卫生类建议(提醒完结 100% 的单)。所以这个功能从「新建一套 backlog 系统」收缩成了「给现有引擎加一个决策维度」——需求被拷问小了,这是好事。

## ticket next:一个有性格的推荐

规则很短,但每条都是拷问出来的立场:

- **focus 置顶**——今天说了要盯的,永远第一。
- **implementing 按进度降序**——离终点最近的先推。85% 的单排在 42% 前面,先减在制品。
- **无半成品才推新坑**——按登记时间降序,新登记的意愿最强。
- **paused / someday 不进池**——挂起就是「别烦我」,只在 `ticket backlog` 里可见。
- **100% 的单不占推荐**——那是「建议完结」的事,开工推荐不跟收尾提醒抢位置。

实现上只有一个 `recommend.py`,CLI、看板卡片、日报建议三处调同一个函数——包括把日报里原有的「聚焦补空」挑单逻辑也换掉了。**一个系统里不能有两套「该做什么」的脑子**,不然今天 CLI 推 A、日报推 B,推荐本身就失去公信力。

跑在真数据上的第一条输出:

```
→ T012 oled-mode-indicator  (desktop) — 11/13 (85%)
  理由: 收尾优先 — 进度已 85%,离完成最近,已停滞 4 天
  开工: ticket focus T012  &&  cd ~/OpenSourceProjects/desktop
```

## 尾声:在别人的在途改动上单独成 commit

一个工程细节。提交时发现工作区里还躺着约 400 行没提交的在途改动,和新功能在同三个文件里交错;更麻烦的是新命令依赖的 `resolve_repo_arg` 本身就是在途改动的一部分——纯挑自己的 hunks 会产出一个 ImportError 的坏 commit。最后的做法:从 HEAD 内容出发重放功能补丁、连同最小依赖函数直接写进 index,工作区一个字节不动;再把这个 commit 用 `git archive` 抽到独立目录跑全量测试,证明它自洽。**「单独成 commit」不是把 diff 拆开就完了,是拆出来的那个 commit 自己要能活。**

## 为什么这事有意思

vibecoding 时代,产出代码的边际成本掉得飞快,稀缺的东西换了位置:**不是「做」,是「决定做什么」**。工具帮你把 todo 存得再全,列表本身不做决策——22 行和 0 行在「我现在该干嘛」面前是等价的。所以这个功能的本质不是 todo 管理,是个**决策收敛器**:把一整板状态收敛成一条带理由的推荐。

另一条更通用:**你对系统问出的问题,就是它下一个功能的规格。**「项目里的 backlog 有哪些」我今天问了一次,明天还会问——与其每次让 agent 现场翻状态库,不如让它变成 `ticket backlog` 一条命令。问题即功能,前提是有人把你那句顺嘴的感慨拷问成 spec。
