> **声明：本文由 AI（Claude Opus 4.6）生成。** 我在 Cursor 中通过对话完成了规则的定义和落地，然后让 Agent 把过程和思考写成了这篇博客。内容经过我的审阅，但行文和结构由 AI 完成。

## 起因

用 Cursor 越多，越发现一个问题：Agent 的默认行为在很多场景下并不理想。它会在你没要求的时候多加抽象、会在你只想修一个 bug 的时候顺手重构半个文件、会在应该先给结论的时候先倒一堆技术细节。

这些不是模型能力问题，而是**行为纪律**问题。

恰好看到 [Andrej Karpathy 的一份 CLAUDE.md](https://github.com/forrestchang/andrej-karpathy-skills)，里面提炼了他观察到的 LLM 编码常见陷阱，简洁有力。结合自己对 Agent 交互体验的思考，我决定把这些习惯固化成 Cursor 的用户级 `alwaysApply` rules，让它在每次对话中都自动生效。

## 两条规则，分工明确

我最终落地了两条独立的全局规则，放在 `~/.cursor/rules/` 目录下：


| 规则文件                                  | 管什么                   |
| ------------------------------------- | --------------------- |
| `personal-interaction-principles.mdc` | 沟通方式：少打断、渐进展示、始终给下一步  |
| `personal-coding-habits.mdc`          | 编码行为：显式确认假设、最简方案、最小改动 |


为什么不合成一条？因为它们的关注点完全不同。交互原则管"怎么跟我说话"，编码习惯管"怎么改我的代码"。分开之后，维护和迭代都更清晰。

## 第一条：交互设计原则

这条规则的核心观点是：**用户体验是所有交互决策的最高准则。** 优先级高于技术偏好、代码整洁度、架构完美度。

五条原则：

### 1. 先讲用户价值，不先讲实现细节

先说这一步对我有什么用，或者接下来我会得到什么结果。技术路径和架构机制不是默认重点，只有影响决策或我追问时才展开。

### 2. 尽量少打断

能自行判断、探索、验证的事，先做，再问。只在存在真实分支、不可逆操作、或关键前提缺失时才提问。需要问的时候，问最关键的一题，不要一次索取整张清单。

### 3. 主动推进

默认承担"下一步该做什么"的判断。先做一轮探索或草拟再汇报，而不是把编排工作留给我。

### 4. 渐进展示

先结论后展开。证据和实现细节按需补充，不要在我还没拿到主线结论时就把所有 caveat 一次倾倒出来。

### 5. 每次反馈都带下一步

不要停在"失败了"或"我在看"。遇到阻塞，同时说当前状态、可能原因、以及准备怎么推进。

## 第二条：编码行为纪律

这条规则直接受 Karpathy 那份 CLAUDE.md 启发，但做了两个关键适配。

### 借鉴了什么

Karpathy 的四条原则里，我融合了三条：

**1. Think Before Coding → 关键假设必须显式确认**

动手之前把理解和假设说出来。存在多种合理解读时列出来让我选，不要默默挑一个。

但这里有一个关键融合点：交互原则说"少打断"，这条说"该问就问"。它们不矛盾——**可以安全推断的事不问（交互原则 #2），但涉及架构方向、不可逆决策、或多种合理解读时，必须先确认（编码原则 #1）。**

**2. Simplicity First → 最简方案优先**

只写被要求的功能。单次使用不做抽象，没人要求"灵活性"就不加。自检标准：一个资深工程师看到会不会说"过度设计了"？

**3. Surgical Changes → 最小改动原则**

只改与任务直接相关的部分。不顺手"改善"旁边的代码。匹配现有风格。发现无关死代码，提一句就好，不要直接删。验证标准：**diff 里每一行都能追溯到我的请求。**

### 没有借鉴的

Karpathy 的第四条 **Goal-Driven Execution**（把任务转化为可验证目标，循环直到通过）——我没纳入，因为我已有一套更完整的实现：[oh-my-cursor](https://github.com/TaoXieSZ/oh-my-cursor) 的 forge 工作流已经内置了 verify-fix loop。

## 为什么用 `alwaysApply` 全局规则

Cursor 的用户级 rules 支持 `alwaysApply: true`，意思是这条规则在**每次对话**中都会自动注入 system prompt，不需要手动激活。

这正好适合"行为纪律"类的规范——你不会希望每次开新对话都要提醒 Agent "别过度设计"、"别一次问我十个问题"。它应该是一个默认人格特质，而不是一个需要手动开启的功能。

## 几个实践感受

1. **中文写规则没问题。** 模型对中文指令的遵守程度和英文一样稳定，而且你自己维护起来更顺手。
2. **反例比原则更有力。** "不要在用户还没拿到结论前先倒一堆细节"比"渐进展示信息"更容易被模型稳定执行。
3. **两条规则的边界要想清楚。** 如果"该不该问"这件事在交互原则和编码原则里各写一次，模型可能会混乱。所以我在编码原则 #1 里显式写了它和交互原则 #2 的关系。
4. **不要写太长。** 规则越长，模型越容易丢条目。每条规则控制在 50 行以内，用反例补充而不是用长段落解释。

## 你可以怎么用

1. 直接 fork [Karpathy 的 CLAUDE.md](https://github.com/forrestchang/andrej-karpathy-skills) 作为起点。
2. 想清楚你自己最在意的 Agent 行为问题是什么——是话太多？改太多？不问就猜？每个人的痛点不同。
3. 把它写成 `~/.cursor/rules/xxx.mdc`，设 `alwaysApply: true`。
4. 保持简短，用反例驱动，定期根据实际体验迭代。
