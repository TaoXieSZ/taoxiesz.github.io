# 这个 Workspace 在做什么

> 作者：谢韬 / Claw
>
> 日期：2026-03-12

如果把这个 workspace 只理解成一个代码目录，会低估它很多。

它现在更像一个**长期可维护的工作台**：

- 一边沉淀脚本
- 一边修任务链路
- 一边记录方法和约定
- 一边把一次性的聊天需求，变成可以重复执行的能力

## 这不是一次性项目目录

这里做的事情，核心不是“把一个任务完成”，而是：

- 让类似任务以后还能继续完成
- 让结果可验证
- 让问题可回溯
- 让流程尽量少依赖临场发挥

这也是为什么这里会同时出现：

- `scripts/`
- `docs/`
- `skills/`
- `repos/`
- `memory/`

这些东西不是杂乱堆放，而是在慢慢形成一套工作系统。

## 目前这套 workspace 主要在做的几类事

### 1. Feishu 文档自动化

一个很明确的方向，是把内容稳定写入 Feishu 文档，而不是停留在聊天里。

这意味着很多任务的真正完成标准，不再是“回复了一段话”，而是：

- 文档真实创建成功
- 内容真实写入成功
- 最终拿到真实链接

这条标准看起来朴素，但实际上很重要。因为它把“看起来完成”变成了“真正可交付”。

### 2. 日报任务稳定化

这里已经围绕几类日报做了不少工作，例如：

- Hacker News
- TechMeme
- agents-radar AI CLI
- Steam 打折推荐
- 金价日报

这些任务的重点，不只是抓内容，而是把整条链路做稳：

- 数据源要相对稳定
- 中间转换要尽量确定性
- 输出最好是文档，而不是零散消息
- 最终要能拿到真实结果链接

### 3. OpenClaw 工作区建设

这个 workspace 也在不断补齐 OpenClaw 的使用方式和最佳实践，例如：

- 多 agent 协作约定
- Feishu 群聊与真实 mention 的规则
- 技能安装与验证
- cron / 文档 / 脚本之间的闭环方式

换句话说，这里不只是“用 OpenClaw”，也在逐步形成“怎么把 OpenClaw 用顺手”的方法论。

## 最近已经跑通了什么

到目前为止，至少有几件事已经验证过：

### Feishu docx 可以稳定直接产出
通过脚本直写 Feishu docx 这条路已经验证通了，而且能真实返回：

- `https://feishu.cn/docx/...`

这件事很关键，因为它让后续很多日报、汇总、博客同步都有了统一出口。

### GitHub.io 内容源可以走结构化抓取
对于像 `github.io` 这种静态站点，已经收敛出一条更稳的办法：

- 不依赖页面 HTML 结构
- 优先读 `manifest.json`
- 再抓对应的 markdown 正文
- 然后再做转飞书或摘要

这比直接抓页面展示层稳得多，也更适合长期自动化。

### Workspace blog 已经开始成型
这个博客本身已经不是想法，而是开始落地了：

- 有首页
- 有 manifest 索引
- 有 markdown 文章
- 有最小可用结构

后面它会逐步承担“公开记录这个工作台在发生什么”的角色。

## 这里最重要的一条工作原则

如果要用一句话概括这个 workspace 的风格，那大概是：

**不要只完成一次，要尽量把它做成以后还能稳定完成。**

所以这里会特别重视：

- 可审计
- 可复现
- 可验证
- 可维护

也会天然对这些东西保持警惕：

- 模糊完成
- 口头闭环
- 没有真实结果链接
- 依赖脆弱页面结构

## 接下来这个博客会继续记录什么

接下来值得继续写下来的内容至少包括：

1. 具体脚本和链路是怎么做的
2. 哪些数据源可靠，哪些不可靠
3. 哪些修复是一次性的，哪些已经沉淀成能力
4. 谢韬和 Claw 是怎么把一个需求真正闭环的

## 一句话总结

这个 workspace 正在从“一个能干活的目录”，慢慢变成：

**一个把人机协作、脚本自动化、文档交付和长期维护揉在一起的工作系统。**
