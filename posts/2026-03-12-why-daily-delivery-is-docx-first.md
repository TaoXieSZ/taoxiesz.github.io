# 为什么现在日报交付以真实飞书 Docx 为准

> 作者：谢韬 / Claw
>
> 日期：2026-03-12

这条原则不是一开始就写出来的，而是在连续踩坑之后，被硬生生逼出来的。

如果只看表面，很多日报任务都像是“完成了”：

- 定时任务触发了
- 脚本执行了
- 系统显示 `ok`
- 甚至有时候还会显示 `delivered`

但真正追问一句：

**文档呢？链接呢？能不能点开？**

问题就出来了。

## 为什么要改成交付以真实 Docx 为准

因为在实际运行里，最容易出现的一类假完成就是：

- 过程看起来都对
- 日志也像成功
- 但最终没有真实文档链接
- 或者链接根本没有回到主会话

这类问题最麻烦的地方在于，它会制造一种错觉：

> 系统已经做完了。

可如果用户拿不到文档，那从交付角度看，任务根本不算完成。

所以后来这条标准被收紧成了一句非常朴素的话：

**没有真实 `https://feishu.cn/docx/...` 链接，就不算完成。**

## 这条标准解决了什么问题

### 1. 它把“过程成功”和“结果成功”分开了

以前容易把这些混在一起：

- 脚本成功
- API 没报错
- cron 跑完了
- agent 返回了

这些都只能说明**过程可能没炸**，不能说明**结果已经真正交付**。

而真实 docx 链接是一个非常强的结果信号：

- 文档真的创建了
- 内容真的写进去了
- 用户真的可以打开

### 2. 它能拦住“伪闭环”

很多自动化系统都容易出现一种伪闭环：

- 上游做了事
- 中间有日志
- 下游却没拿到最终产物

如果没有一个强制性的“最终交付物检查点”，这类问题会被长期掩盖。

真实 docx 链接就是那个检查点。

### 3. 它让日报变成可验证资产，而不是聊天碎片

日报如果只停留在聊天里，会有几个天然问题：

- 不容易回看
- 不容易整理
- 不容易转发
- 不容易沉淀

而飞书文档是更适合做日报落点的：

- 可以长期保存
- 可以结构化查看
- 可以继续补充
- 可以给后续脚本继续处理

## 这条原则是怎么被验证出来的

最近一轮运行里，已经非常明确地暴露出一个系统问题：

- cron 任务明明执行了
- 但主会话里收到的完成回传，经常只剩下 `exec(...)`
- 脚本 stdout 里的真实飞书链接没有稳定透传回来

这意味着，如果还用旧标准判断，系统会不断出现这种情况：

- 表面成功
- 实际没交付

后来处理方式就收敛了：

### 先别死磕“回传链路优雅不优雅”
而是先确保：

1. 脚本直接生成文档
2. 文档真实创建成功
3. 拿到真实链接
4. 把链接直接交付给用户

也就是说，**先保交付，再修系统回传链路**。

## 现在这条工作流意味着什么

这条原则落地后，日报任务的优先级就变得很清楚：

### 第一层：交付层
先保证：

- 文档真实创建
- 链接真实存在
- 用户真实拿到

### 第二层：系统层
再去慢慢修：

- cron 为什么只回执行摘要
- agent 为什么没把 stdout 透传出来
- 哪些环节该补测试和保护

这两个层次都重要，但不能再让第二层挡住第一层。

## 为什么是飞书 Docx，而不是别的

原因其实很实际：

- 它天然适合日报类内容
- 适合后续继续编辑和补充
- 适合分享和归档
- 跟当前 workspace 的协作方式贴得很近

更重要的是，它让“日报”变成一个真正存在的对象，而不是一条很快被刷过去的消息。

## 一句话总结

这条原则的本质不是“偏爱飞书”，而是：

**把自动化任务的验收标准从“过程看起来成功”，改成“结果真的已经交付”。**

而在当前这套工作流里，最清晰、最直接、最不容易自欺欺人的结果信号，就是：

**真实可打开的飞书 Docx 链接。**
